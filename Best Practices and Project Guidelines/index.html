Use proper Doctype

Doctype is a must for every webpage. And it should be first thing in the page source above <html> and NOTHING should be above it, be any HTML markup, CSS rules, JS code or any comments. Otherwise, the browser will render the page in quirks mode and may cause some unexpected issues.

For HTML5, which is the current standard for web pages, use <!DOCTYPE html>

Additionally, every page should have <html>, <head>, <body> & <title> tags.

Make efficient use of selectors

Keep specificity to a minimum

Specificity is the order of the CSS rules in which they will be applied.

A CSS selector with higher specificity overrides the one with lower specificity. Only the conflicting properties are overridden by the higher specific selector. If the selectors have equal specificity, the one which was declared at the last is the winner.

  1. Ids selector has a specificity of 100
  2. Classes, pseudo-classes, attributes selectors have a specificity of 10
  3. Elements, pesudo-elements selectors have a specificity of 1
  4. Universal selector(*) has a specificity of 0
Rule Specificity
p 001
div p 002
.name p 011
#content 100
body #content .name p.title a:hover 133
body [id=content] .name p:first-line 023
body #content p:not(.title) 112
body #content div.name p.title { color: red; font-size: 15px; } .name .title { color: red; font-size: 15px; }

Note:

  • :not() is a pseudo-class but it doesn't add any specificity on its own, but what is inside its parantheses does.
  • Inline styles override any order of specificity. Think of its value as 1000.
  • Inline styles can only be overridden by the use of !important. Think of its value as 10000.

Avoid using #id

Id has a specificity of 100, thus to override this selector, we are forced to use id again increasing the complexity and decreasing the performance. Id should be introduced only if absolutely necessary.

#quantity { border: 1px solid gray; font-size: 15px; } .quantity { border: 1px solid gray; font-size: 15px; }

Avoid using !important

Using !important is always a bad idea for the very same reason as using Id. Try increasing specificity to achieve the same result by adding classes or element tags.

To override !important, we will need another !important with greater specificity. We should always try to remove the !important from our code.

.label { color: red; font-size: 15px; } ul li { color: green !important; font-size: 18px !important; } .label { color: red; font-size: 15px; } ul li.label { color: green; font-size: 18px; }

Using a grid system eliminate the need of new styles and helps to maintain consistency through the code. B. Bad Practices and Better Design

Avoid using global variables

Variables declared outside of a function are attached to the global scope. Additionally, variables used without declaring it first, i.e. without using var keyword, whether it is inside or outside any function makes it a global variable.

Global variables are extremely risky as they can used or modified by any part of the code or some other third party's code, either intentionally or un-intentionally.

Always declare the variables locally by using var keyword. Namespacing, closures, self-executing immediate functions are helpful in resolving clashing of variables and minimize global variables clutter.

var pageId = 100; function init() { var pageName = 'home'; pageId = 200; count = 1; console.log(pageId); // 200 } init(); console.log(pageId); // 200 console.log(count); // 1 console.log(pageName); // not defined var MYAPP = MYAPP || {}; MYAPP.pageId = 100; MYAPP.init = function() { var pageName = 'home'; var pageId = 200; var count = 1; console.log(pageId); // 200 } MYAPP.init(); console.log(pageId); // not defined console.log(MYAPP.pageID); // 100 console.log(count); // not defined console.log(pageName); // not defined

Declare all of the variables at the starting of the functions and preferably assign some initial value. Hoisting?

Never declare variables inside a block

Always use === instead of ==

Using === make sure that you are comparing the values of similar types. This will prevent the some weird quirks of type casting when '==' is used. There is no need to compare values of different types and if you found such scenarios, re-visit the logic behind it.

Use caution while converting == to === when

  • Comparing string and number, use parseInt() / parseFloat() to convert the string into number before comparsion
  • Comparing null and undefined, either use both or better use truthy/falsy values for comparison

eval() === evil()

Never use eval() as it execute the arbitrary string as a code, raising the risk of serious security issues.

DOM Manipulation

DOM manipulation is a heavy task and has a lot of performance impact.

Optimize the query for selectors. Start with id attribute if possible and then move your way down to elements, class and so on.

Multiple id selector in a single query is useless. Also, query the id selector alone and chain other selectors.

$('#first #second')... $('#first .class1')... $('#second')... $('#first').find('.class1')...

While selecting some DOM node which can be used later, make sure to Cache it. This will save time to re-select the DOM when querying it next time.

$('#first').addClass('one'); $('#first').addClass('two'); $('#first').removeClass('three'); var $first = $('#first'); $first.addClass('one'); $first.addClass('two'); $first.removeClass('three');

Chaining can be also be used to invrease performance.

var $first = $('#first'); $first.addClass('one'); $first.addClass('two'); $first.removeClass('three'); var $first = $('#first'); $first.addClass('one two').removeClass('three');

Event Binding

Event binding can be stacked, i.e. if there are multiple event handlers for same event on the same DOM node, event will fire for each of the stacked event handler.

Avoid Event binding in a loop, function or callbacks. As each time the function will be called event binding will be stacked.

function random() { $('.link').on('click', function() { // handler }); } function random() { } $('.link').on('click', function() { // handler });

Try delegating the events so that they can still work for dynamically added DOM nodes like that added after ajax call.

$('.link').on('click', function() { // handler }); $('body').on('click', '.link', function() { // handler });

HTTP requests

HTTP requests should be kept to a minimum as they have overhead of moving back and forth to server. This includes requests for images, stylesheets, js and ajax calls. Browsers also cache these files

Minifiy and concatenate css and js files. Use sprties to reduce image requests.

Avoid unneccessary ajax calls.

Reduce Redundant code

Modularize your code. Try to break a large and/or nested function into multiple smaller functions.

Take out similar code out of if-else block.

function doSomething() { // some code if (conditon) { // executes when condition evaluates to true . . . // similar set of statements statement 1; statement 2; statement 3; } else { // executes when condition evaluates to false . . . // similar set of statements statement 1; statement 2; statement 3; } } function doSomething() { // some code if (conditon) { // executes when condition evaluates to true . . . } else { // executes when condition evaluates to false . . . } // move the statments out of if-else block statement 1; statement 2; statement 3; }

Try to bundle similar set of instruction into a function which can be used on later wherever needed.

Difference between Abstract Equality (==) and Strict Equality (===) ?

  1. === compares TYPE and value
  2. == compares ONLY value, not the type

Statement 1 is OK, but statement 2 is very much flawed. '==' also checks for the type of the operands. The difference is that '===' return false if the type do not match while '==' coerce the operands into similar types and then return the result similar to '==='.

A simplified version of Abstract Equality Comparison (a == b) for primitive values:

  1. If *Type(a) is same as Type(y), return the result of a === b
  2. If a is null and b is undefined (or vice versa), return true
  3. If either a or b is Boolean or String, convert them into Number and repeat from Step 1
  4. Return false

*Type(a) refers to the language type and it is different from typeof a. The ECMAScript language types are Undefined, Null, Boolean, String, Symbol, Number, and Object.

Truthy/Falsy Values:

There are 6 falsy values:

  1. false
  2. 0
  3. ''
  4. null
  5. undefined
  6. NaN

Any value that is not a falsy value, is a truthy value.

This inlcudes '0', 'false', {}, [], new Boolean(false), etc.

The condition of if (cond) {} block checks for truthy value, not the boolean true value.

if (value !== null && value !== undefined && value !== '') { } if (value === null || value === undefined) { } if (value) { // a shorthand but remember this will check for all falsy values // so if value can be 0 or false, this should not be used } if (!value) { }
